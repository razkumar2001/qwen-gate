## Part 1: Performance Audit

### Finding P1: O(n²) Content Filtering on Accumulated Text Per Chunk

| Field | Detail |
|---|---|
| **File** | `src/utils/contentFilter.ts`, lines 7–40 (called from `src/routes/chatStreamingHelpers.ts`) |
| **Category** | CPU |
| **Severity** | **High** |

`filterContent(raw)` is invoked on every streaming chunk. However, it receives the **full accumulated text** (not just the delta) and re-processes it from scratch each time. The `while (true)` loop calls `text.match()` which scans from the beginning of the string on every iteration. As the response grows, work per chunk grows linearly, making total work O(n²).

**Impact:** For a 10,000-char response in 500 chunks, ~2.5M chars processed total instead of 10,000. Latency per chunk increases linearly, degrading time-to-last-token for long responses.

**Fix:** Process only the delta (new content since last chunk). Maintain a `contentFilterState` with last-processed position and incrementally append/process. Handle think-tag boundaries spanning chunks with a ~200 char carry-over buffer.

### Finding P2: Dynamic RegExp Compilation Inside Hot Loop

| Field | Detail |
|---|---|
| **File** | `src/utils/contentFilter.ts`, ~line 18 |
| **Category** | CPU |
| **Severity** | **Medium** |

Inside the `while (true)` loop, a new `RegExp` is constructed every iteration:

```ts
const endPattern = new RegExp(`</${endTagName.replace(/[\s>].*/, '')}>`, 'i');
```

**Impact:** Adds microseconds per iteration, compounding with P1. Measurable on long responses with multiple think blocks.

**Fix:** Pre-compile patterns for the small set of possible tag names (`think`, `thinking`, `thought`).

---

### Finding P3: Excessive String Allocations in Filter Pipeline

| Field | Detail |
|---|---|
| **File** | `src/utils/contentFilter.ts`, lines 40–85 |
| **Category** | Memory / CPU |
| **Severity** | **Medium** |

After the think-tag extraction loop, a chain of full-text transformations runs per chunk:
- `text.split('\n\n')` → array of all paragraphs
- Per-paragraph: `split('\n')`, `filter()`, `join('\n')`
- `cleanParagraphs.join('\n\n')` → reassembles string
- Three additional `text.replace(...)` passes
- `stripToolCallArtifacts(text)` → another full-text pass

**Impact:** High GC pressure during streaming. 100KB response × 500 chunks = hundreds of MB of intermediate strings. Under concurrent requests, GC pauses stall all active streams.

**Fix:** Single-pass filtering, or apply only to delta (fixes P1 simultaneously).

---

### Finding P4: Stateful Global RegExp Shared Across Concurrent Calls

| Field | Detail |
|---|---|
| **File** | `src/utils/thinkTagStripper.ts`, line 36 |
| **Category** | Concurrency / Correctness |
| **Severity** | **Medium** |

```ts
export const QWEN_THINK_TAG_PATTERN = /<\/?(?:think(?:ing)?|thought)(?:\s[^>]{0,100})?\/?>/gi;
```

Uses `g` flag, making `lastIndex` stateful. If any caller uses `.test()` or `.exec()` without resetting `lastIndex`, concurrent requests will get stale positions.

**Impact:** Intermittent bugs — think tags partially stripped or not stripped at all, depending on interleaving.

**Fix:** Factory function returning fresh regex instances, or reset `lastIndex` at every call site.

---

### Finding P5: Unbounded In-Memory Request Log Store

| Field | Detail |
|---|---|
| **File** | `src/services/logStore.ts` |
| **Category** | Memory |
| **Severity** | **Medium** |

`RequestLogStore` keeps request/response records in memory with no visible eviction policy or max size cap. Includes full content previews.

**Impact:** Memory grows without bound on long-running servers. Hundreds of MB over time under sustained load → OOM or aggressive GC.

**Fix:** Add max entry count (e.g., 1000) with ring-buffer/LRU eviction. Drop response bodies after finalization.

---

### Finding P6: Per-line 18-branch Regex in Hot Path

| Field | Detail |
|---|---|
| **File** | `src/utils/thinkTagStripper.ts`, lines 1–33 |
| **Category** | CPU |
| **Severity** | **Low** |

`isThinkingLine` tests each line against a regex with 18 alternation branches. Most streaming content is not thinking text, so most tests return `false` after full scan.

**Impact:** Minor per-line, but multiplied by P1's re-processing of full text per chunk.

**Fix:** Quick first-char/first-word check before full regex. Or fix P1 to make this negligible.

## Part 3: Bug Investigation

### 🔴 B1: Timer Leak in Streaming Loop

| Field | Detail |
|---|---|
| **File** | `src/routes/streamLoop.ts`, lines 49-52 |
| **Severity** | **Critical** |

Each iteration creates a `setTimeout` via `Promise.race` for idle timeout detection. When `reader.read()` resolves first (every chunk), the timer is **never cleared**. N orphaned timers accumulate per stream.

```typescript
const readResult = await Promise.race([
  reader.read(),
  new Promise<never>((_, reject) => {
    setTimeout(() => reject(new Error('idle timeout')), IDLE_TIMEOUT_MS);
  }),
]);
```

**Impact:** Memory leaks, event loop congestion, potential "too many open files" errors under sustained load.

**Fix:** Store timer reference and clear with `.finally()`:
```typescript
let timeoutId: NodeJS.Timeout;
const readResult = await Promise.race([
  reader.read().finally(() => clearTimeout(timeoutId)),
  new Promise<never>((_, reject) => {
    timeoutId = setTimeout(() => reject(new Error('idle timeout')), IDLE_TIMEOUT_MS);
  }),
]);
```

### 🔴 B2: Wrong parentId Passed to Waiter Session

| Field | Detail |
|---|---|
| **File** | `src/services/sessionPool.ts`, lines 152-158 |
| **Severity** | **Critical** |

When a waiting request gets a **new chatId** from `createSession()`, it receives the **old session's parentId** — which doesn't exist in the new chat.

**Impact:** API errors from Qwen (parent message not found), broken conversation threads, failed requests for queued users.

**Fix:** Pass `null` as parentId for new waiter sessions.

---

### 🟠 B3: In-Flight Counter Leak on Error Path

| Field | Detail |
|---|---|
| **File** | `src/services/sessionPool.ts`, lines 94-104 |
| **Severity** | **High** |

`incrementInFlight()` is called unconditionally, but `decrementInFlight()` is only called conditionally in the catch block. During multi-account retries, failed accounts retain inflated counters.

**Impact:** Incorrect load balancing, potential account starvation, degraded service.

**Fix:** Use try-finally pattern to ensure decrement always matches increment.

---

### 🟠 B4: Race Condition in Browser Context Creation

| Field | Detail |
|---|---|
| **File** | `src/services/playwright.ts`, lines 202-244 |
| **Severity** | **High** |

`contextCreationInFlight` check and `accountContexts.has()` check are not atomic. Two concurrent calls for the same email both pass checks, both create contexts, second overwrites first — **first context leaked** (never closed, refresh interval still running).

**Impact:** Browser context leaks, duplicate refresh intervals, memory/file descriptor leaks.

**Fix:** Add `accountContexts.has(email)` check before entering creation path; make creation atomic.

---

### 🟠 B5: Cached Cookies Shared Across All Accounts

| Field | Detail |
|---|---|
| **File** | `src/services/playwright.ts`, lines 77-93 |
| **Severity** | **High** |

`getCookies()` without email returns cached cookies from the **first account** in the map, regardless of which account should be used. Global `cachedCookies` variable shared across all callers.

**Impact:** Wrong authentication, security issue (cross-account cookie usage), intermittent 401 errors.

**Fix:** Cache cookies per-account using a `Map<string, { cookies, timestamp }>`.

---

### 🟡 B6: Byte-Level Truncation Splits UTF-8 Characters

| Field | Detail |
|---|---|
| **File** | `src/routes/compressToolResult.ts`, lines 7-24 |
| **Severity** | **Medium** |

`Uint8Array` slicing at arbitrary byte offsets splits multi-byte UTF-8 characters. `TextDecoder({ fatal: false })` produces replacement characters. Truncation message mixes char count with byte count.

**Fix:** Adjust split points to respect UTF-8 character boundaries.

---

### 🟡 B7: Inconsistent `.js` vs `.ts` Import Extensions

| Field | Detail |
|---|---|
| **Files** | `sessionPool.ts`, `auth.ts`, `modelRouter.ts` |
| **Severity** | **Medium** |

Some files import `logStore` with `.js` extension, others with `.ts`. Works in tsx but may fail in compiled output.

**Fix:** Standardize all imports to `.js` extensions (ESM convention).

---

### 🟡 B8: Throttled Accounts Still Returned

| Field | Detail |
|---|---|
| **File** | `src/services/accountManager.ts`, lines 450-461 |
| **Severity** | **Medium** |

`getTokenWithAccount` returns throttled accounts if they have a valid token, defeating the throttling mechanism.

**Impact:** Throttled accounts continue to be used, causing repeated failures.

**Fix:** Return `null` for throttled accounts.

### 🟡 B9: Empty Cookies Returned Silently

| Field | Detail |
|---|---|
| **File** | `src/services/playwright.ts`, lines 68-93 |
| **Severity** | **Medium** |

When no accounts exist, `getCookies()` returns empty string instead of throwing/returning null. Downstream code uses empty cookies → silent auth failures.

**Fix:** Return `null` or throw to make the failure explicit.

---

### 🟡 B10: No Session Limit Enforcement

| Field | Detail |
|---|---|
| **File** | `src/services/sessionPool.ts` |
| **Severity** | **Medium** |

`MAX_WAITING = 10` is defined but no `MAX_SESSIONS` limit exists. Under high concurrency, unlimited sessions can be created.

**Impact:** Resource exhaustion, memory pressure, degraded performance.

**Fix:** Add `MAX_SESSIONS` constant and queue excess requests.
